## Problem Statement

택배 상태를 능동적으로 확인하고 싶을 때, 매번 운송장 번호를 찾아서 택배 조회 사이트에 입력해야 한다. 중요한 택배의 경우 이 행위를 하루 10회 이상 반복하게 되고, 이 반복 자체가 피로감을 유발한다. 기존 카카오톡/문자 알림은 "시스템이 알려주는 것"이지 "내가 확인하고 싶을 때 즉시 확인하는 것"이 아니다.

## Demand Evidence

- 창업자 본인의 경험: 중요한 택배를 하루 10회 이상 확인하며 반복 행위에 피로를 느낌
- 외부 수요 검증은 아직 없음. 출시 후 검증 필요
- 세컨드 오피니언 분석: 이 행위는 정보 필요가 아닌 "불안 해소 행동"에 가까움. 5분 전과 상태가 같은데도 확인한다. 필요한 것은 "변한 거 없어, 걱정 마" 또는 "변했어, 봐"를 항상 보여주는 지표

## Status Quo

현재 사용자 워크플로우:
1. 운송장 번호 확인 (문자, 주문내역 등에서 찾기)
2. 택배 조회 사이트 접속
3. 운송장 번호 입력
4. 조회 버튼 클릭
5. 현재 상태 확인

이 5단계를 하루에 10회 이상 반복. 기존 대안(스마트택배, 배송지키미, CJ대한통운 앱)도 "앱을 열어야" 확인 가능. 카카오톡/문자 알림은 수동적 수신이지 능동적 확인이 아님.

## Target User & Narrowest Wedge

**타겟:** 중요한 택배(중고거래, 해외직구, 선물 등)가 올 때 하루에도 여러 번 능동적으로 확인하는 iPhone 사용자.

**Narrowest Wedge:** 앱을 열 필요 없이 잠금화면/Dynamic Island만 보면 택배 상태가 바로 보이는 것. 운송장 번호 한 번 입력하면 끝.

## Constraints

- Dynamic Island 탑재 기기(iPhone 14 Pro+) 한정
- 한국 택배사 공식 개인 개발자 API 없음, 서드파티(스마트택배) 또는 크롤링 의존
- Live Activity는 Apple의 ActivityKit 제약 내에서만 동작 (업데이트 빈도, 크기 제한)
- 1인 개발

## Premises

1. 사람들은 택배 상태를 능동적으로, 반복적으로 확인하고 싶은 욕구가 있다
2. Dynamic Island는 앱을 열지 않고 즉시 확인하는 UX에 최적의 인터페이스다
3. 수익은 구독제(Waito Plus ₩2,900/월 or ₩19,900/년). 무료: 기본 트럭 1종 + Live Activity 1개. 구독: 프리미엄 스킨 전체 + Live Activity 2개 + 데코 모드(택배 없을 때도 트럭이 달리는 꾸미기 전용 모드). 추적 자체는 무료 무제한
4. 택배 API는 서드파티/크롤링에 의존해야 한다
5. iPhone 14 Pro 이상이 타겟 기기다

## Cross-Model Perspective

독립적인 Claude 서브에이전트의 cold read:

**Steelman:** Waito는 택배 추적을 "능동적 행위"에서 "수동적 인식"으로 바꾸는 유일한 앱. Dynamic Island라는 모든 앱 위에 보이는 UI에 트럭을 올려서 한 번의 시선으로 해결. 추적 기능이 설치를 정당화하고, 감성이 유지와 결제를 정당화.

**Key Insight:** 하루 10번 확인은 정보 필요가 아니라 불안 해소 행동. "불안을 위해 만들어라, 정보를 위해가 아니라."

**Challenged Premise:** 수익 모델. Pixel Pals의 펫은 항상 살아있지만, 목적지 없는 트럭은 의미가 없다. → 이 도전을 반영하여 구독자 전용 "데코 모드" 추가: 택배가 없을 때도 트럭이 Dynamic Island를 달리는 꾸미기 전용 모드. 구독 가치 = 프리미엄 스킨 + Live Activity 2개 + 데코 모드. 택배 없는 기간에도 구독 유지 이유 확보.

**48h Prototype:** 서버 없이 가짜 데이터로 Live Activity + 트럭 애니메이션만 구현, TestFlight 5-10명 배포, 3일 후 "평소 택배 앱을 덜 열었나요?" 질문으로 검증.

## Approaches Considered

### Approach A: 현재 구조 유지 + StoreKit 구독 연동 (CHOSEN)
기존 코드/구독 구조 유지. SubscriptionManager를 StoreKit 2 실제 구독으로 연동. 데코 모드 추가.
- Effort: S (human: ~3일 / CC: ~30분)
- Pros: 기존 코드 거의 그대로, 가장 빠른 출시
- Cons: StoreKit 2 연동 + App Store Connect 설정 필요

### Approach B: 감성 플랫폼
시즌 한정 스킨 + 도착 효과 + 사운드팩 + 귀환 모드.
- Effort: M
- Pros: 반복 구매 유도, 택배 없는 기간에도 감성 유지
- Cons: 에셋 제작 비용, 시즌 운영 부담

### Approach C: 48시간 검증 프로토타입
서버 없이 가짜 데이터, Live Activity만 TestFlight 배포.
- Effort: S
- Pros: 가설 검증 먼저
- Cons: 이미 상당 부분 구현됨, 출시 지연

## Recommended Approach

**Approach A.** 이미 코드가 상당히 진행되어 있고(3커밋 + 6파일 미커밋 변경), 기존 구독 구조를 유지하면서 StoreKit 2로 실제 연동하면 출시에 가장 가깝다. Approach B의 시즌 스킨은 출시 후 확장.

구체적 변경:
1. SubscriptionManager를 StoreKit 2 자동 갱신 구독으로 연동 (현재는 UserDefaults 디버그 토글)
2. 데코 모드 추가: 구독자가 택배 없을 때 트럭이 Dynamic Island를 달리는 꾸미기 전용 Live Activity
3. 기존 TruckConfig/TruckShape/TruckStyle/TruckColor 시스템 유지, isPremium/canUse 판별 로직 그대로
4. 구매 복원 플로우 추가
5. 무료: 기본 트럭 + LA 1개, 구독: 프리미엄 스킨 + LA 2개 + 데코 모드 (기존 구조와 동일)

## API Dependency & Fallback

택배 조회 API는 이 앱의 핵심 인프라이며, 1인 개발 체제에서 API 장애 = 앱 전체 기능 중단이므로 반드시 대비해야 한다.

- **Primary:** 스마트택배 API (tracker.delivery) — 무료 티어 한도 및 상용 이용약관 확인 필수.
- **Decision Gate:** App Store 제출 전에 tracker.delivery 상용 이용 가능 여부를 확정해야 한다. 불가 시 대안: (1) AfterShip 유료 티어 (월 ~$9), (2) 자체 크롤링 서버. 이 결정 없이 App Store 제출 금지.
- **Fallback:** API 장애 시 캐시된 마지막 상태를 표시하고 "마지막 확인: X분 전" 타임스탬프를 보여줌. 사용자에게 "일시적 조회 불가" 상태를 명시적으로 노출.

## Privacy & Accessibility

- **개인정보:** 운송장 번호는 PII에 준하는 데이터. 서버에 저장 시 암호화 필수. App Store 제출 시 개인정보 처리방침 필요. 데이터 보존 기간 정책 수립 필요 (배송 완료 후 30일 자동 삭제 등).
- **접근성:** Dynamic Island의 트럭 애니메이션은 시각적 요소이므로 VoiceOver 사용 시 "배송 중, 간선 상차 완료" 같은 텍스트 대안 제공. 잠금화면 뷰의 프로그레스 바도 접근성 레이블 추가.

## Open Questions

1. 택배 없는 기간에 사용자 이탈을 어떻게 막을 것인가? (귀환 모드, 일상 트래커 등 — 출시 후 데이터로 판단)
2. App Store 심사에서 Live Activity 사용이 리젝되지 않을지 (Pixel Pals 선례 있으므로 가능성 높음)
3. 트럭 스킨 에셋을 누가 만드는가? (직접 제작 vs 외주 vs AI 생성)

## Success Criteria

- App Store 출시 완료
- 첫 달 다운로드 1,000건 이상
- 일회성 스킨 구매 전환율 5% 이상
- App Store 평점 4.5 이상
- "택배 앱을 덜 열게 됐다"는 리뷰 최소 3건

## Distribution Plan

- App Store 배포 (iOS only)
- TestFlight으로 베타 테스트 선행
- 서버는 별도 호스팅 필요 (현재 localhost, 배포 시 실제 서버 URL 필요)
- CI/CD: Xcode Cloud 또는 GitHub Actions로 TestFlight 자동 배포 고려

## Dependencies

- 스마트택배 또는 대체 택배 조회 API 확보
- Apple Developer Program 등록 (App Store 배포용)
- 서버 호스팅 환경 구축 (API 서버 배포)
- StoreKit 2 인앱 구매 설정 (App Store Connect)

## The Assignment

TestFlight으로 주변 5-10명에게 현재 버전을 배포하세요. 서버는 임시 배포(Railway, Fly.io 등)로 돌리고, 실제 운송장 번호를 입력하게 해서 3일간 사용하게 한 뒤 한 가지만 물어보세요: "택배 조회 사이트/앱을 평소보다 덜 열었나요?" 이 답이 Yes면 제품이 있는 겁니다.

## What I noticed about how you think

- "문자나 카톡으로 발송해주는 건 사용자가 확인하는 게 아닌 확인을 받는 수동적인 상황" — 능동적 확인과 수동적 알림을 구분하는 이 프레이밍은 날카롭습니다. 대부분의 사람들은 이 차이를 구분하지 못합니다.
- 수익 모델 도전에 대해 "구독 대신 일회성으로 해라"라고 즉각 결정한 것. 서브에이전트의 지적을 수용하되, 감성 커스터마이징이라는 방향 자체는 유지하면서 결제 구조만 바꾼 것이 좋은 판단입니다.
- 본인이 타겟이라고 솔직하게 말한 것. 가장 좋은 제품은 만드는 사람이 첫 번째 사용자인 경우가 많습니다.
